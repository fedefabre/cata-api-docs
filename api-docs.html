CATA Render Service (4.0.0)

Download OpenAPI specification:

Federico Fabre: federicofabre@gmail.com

Asynchronous, job-based render service for the CATA virtual try-on app.

Consumed by the Core Backend as the sole external caller. The Flutter client does not call this API directly — it goes through Core.

Ownership boundary

Domain Core Backend Render Service
User identity, auth issuance ✅ source of truth (Supabase) validates JWT only
Product / garment metadata (catalog, attributes, lifecycle) ✅ source of truth thin reference (garment_id only)
Avatar metadata + active avatar state ✅ source of truth thin mirror (user_id, active_avatar_id, avatar_v)
Source asset bytes (uploads) ✅ stores in CDN reads by URL/ID
Jobs, processed assets, renders, recipe runs — ✅ source of truth
Render-specific state (versions, recipes, lifecycle) — ✅ source of truth

The Render Service holds thin references, not authoritative copies, of Core-owned entities. Garment and avatar metadata lookups beyond what is needed for job execution and version-aware retrieval go to Core, not Render.

URL immutability and caching

Every CDN URL returned by this API is immutable — once an asset is produced at a given version, its URL never changes. New versions are published at new URLs; old URLs continue to serve the bytes they were minted with. Callers may cache any URL indefinitely without coordination.

The only thing that changes over time is which URL is returned by the /assets/.../latest endpoints. A caller treating those endpoints as a short-TTL cache lookup is the recommended pattern.

Common flows

Garment upload: Core uploads source bytes to its CDN, then POST /jobs { type: "garment_processing", ... }. Polls GET /jobs/{id} until status: "completed"; reads the processed asset URL from outputs.

Fitting Room load: Core calls POST /assets/garment-avatar-render/batch with the user and the list of garments. The endpoint returns the latest URL per pair, or not_found for pairs that haven't been rendered yet. For misses, Core fires individual or batched render jobs and tells Flutter "rendering".

Caching responsibility lives on the device, not in Core. Every CDN URL returned is immutable (see above), so Flutter's standard image cache (e.g. cached_network_image) keyed by URL handles "fitting room feels instant on revisit" with no infrastructure work. Core should call the /batch endpoint on every fitting-room load; the endpoint is sized for it.

Avatar change: Core fires POST /jobs { type: "avatar_backfill", ... }. Render fans out child render jobs for the user's wardrobe; pending renders against the previous avatar version are marked superseded. Core polls the parent job for the children list.

See architecture.md for the full production architecture, supersession mechanics, and decision rationale.

Jobs

Create, poll, and retry render-service jobs.

Create a job

Single endpoint for all seven job types. Payload is discriminated by type. Returns 200 on idempotency hit (existing job), 202 on new job.

Authorizations:
(SupabaseJWTServiceKey)
Request Body schema: application/json
required
type
required
string (JobType)
Enum: "garment_processing" "avatar_processing" "garment_avatar_render" "garment_rerender" "avatar_backfill" "garment_reprocess" "avatar_reprocess"

Seven job types across three worker pipelines. Five product-facing; avatar_reprocess and garment_reprocess are triggered internally on recipe-version bumps.

idempotency_key
required
string

user_id
required
string

Must match JWT sub claim.

required
InputsGarmentProcessing (object) or InputsAvatarProcessing (object) or InputsGarmentAvatarRender (object) or InputsAvatarBackfill (object)

Type-specific payload. See per-type schemas.

object (ViewAssumptions)

Output-mode flags. Participates in the render cache key.

priority
integer

Create multiple jobs in one call

Symmetric with POST /jobs but accepts an array. Used after a /assets/garment-avatar-render/batch call to fire render jobs for the misses. Per-item idempotency keys still apply individually.

Returns one ack per item, in the same order. Partial success is possible — a per-item status: "error" on validation failures does not block the rest of the batch.

Bulk lookup of latest renders

The Fitting Room read path. Returns the latest valid render for each (user_id, avatar_id, garment_id, view_assumptions) query in a single round trip. Sized for the common case where a Fitting Room load needs URLs for many garments against the user's active avatar.

Per-query result is either found (with the render) or not_found (caller may fire a garment_avatar_render job for that pair, or use POST /jobs/batch to fire several at once). Read-only — this endpoint never creates jobs.
